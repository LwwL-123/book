## 功能

- 全站各业务的登录用户和未登录用户的点赞，稿件、评论、弹幕等22种业务类型；
- 为其他业务方提供对象的点赞计数查询和用户是否点赞查询；
- 展示用户的点赞列表；
- 展示对象的点赞人列表；
- 展示关注人是否点赞了某个对象；
- 空间页展示用户发布的内容的总点赞数；

### 点赞、点踩流程

![image-20220705141241988](https://picture-1258612855.cos.ap-shanghai.myqcloud.com/20220705141242.png)

点赞逻辑(未登录用户点赞逻辑与其类似)：

点赞、点踩须互斥。须结合当前点赞状态和当前用户的点赞请求，分别做状态更新和计数更新。并适用于文章、评论、tag当前的点赞和计数逻辑。

更新的数据库包括：用户点赞表、稿件被点赞计数表；

更新的缓存包括：用户点赞列表、稿件被点赞列表、稿件被点赞计数表；

| 点赞流程                | 说明                                                         |
| :---------------------- | :----------------------------------------------------------- |
| 步骤1：管控状态校验     |                                                              |
| 步骤2：点赞类型校验     | 点赞对象的类型校验，目前支持22种类型，参考[点赞QuickStart](https://info.bilibili.co/pages/viewpage.action?pageId=23123010&src=contextnavpagetreemode)； |
| 步骤3：点赞状态校验     | 从db查点赞的状态，与用户的点赞行为做校验，确定是点赞、取消点赞、重复点赞、点赞反转(之前是点踩状态)；如果是重复点赞，需要发送消息给job； |
| 步骤4：发送消息给job    | 发送点赞的消息                                               |
| 步骤5：job校验msg消息   | 点赞对象的类型校验                                           |
| 步骤6：写入db           | 校验点赞状态，参考步骤3；更新db的点赞状态；                  |
| 步骤7：聚合稿件的点赞数 | 为了防止热门稿件对db造成写压力过大，对稿件的计数聚合后再写入db； |
| 步骤8：更新缓存         | 稿件状态；用户点赞列表；稿件点赞人列表；                     |
| 步骤9：发送点赞消息     | 用于前端更新稿件的点赞数                                     |
| 步骤10：拜年祭相关      | 增加主稿件的点赞数，修改对应的缓存等；                       |

### 用户被点赞计数

![image-20220705141339577](https://picture-1258612855.cos.ap-shanghai.myqcloud.com/20220705141339.png)

**架构说明:**

1. 点赞行为走job的原因:

​     消息队列削峰 针对拜年祭这种场景 提高稳定性 同一个用户的点赞/取消点赞操作 顺序处理 避免事务冲突 同样可以聚合计数 减少db压力 

   \2. 判断是否点赞不全用zscore的原因:

​     zscore会导致redis qps报警 单实例3w多 换zrange的话 qps 7k 

   \3. 点赞多job:

​      点赞以messageID发送消息 使得对某个对象的点赞都交给一个job去处理

   \4. 异步发点赞消息的原因:

​      同步发的时候pub会超时 所以改成了异步发送

​    \5. 用户被点赞列表：

​     为了 1.防止消费binlog不幂等(原来是incr累加计数) 2.防止双机房计数不一致(key过期后回源) 3.减轻db压力(恶意刷用户个人空间) 将用户发布的所有稿件、动态等的点赞计数全量加载到redis里，按照每个类型一个key存储。
